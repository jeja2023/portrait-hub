## 1. 现状审查结论

重建前旧控制台位于 `frontend/console/`，约 7300 行原生 JS/CSS，无构建工具、无框架、
无模块系统：23 个 JS 文件按 `<script defer>` 顺序加载，依赖全局函数互相调用，
HTML 全部由字符串模板拼接。审查发现的核心问题：

### 1.1 JSON 泛滥是系统性的

- 全站共 **29 处** `json-view data-viewer` 区域（core.js 11 处、access.js 8 处、
  governance.js 10 处），几乎每个页面的最终输出都是 JSON 面板。
- **核心业务结果只有 JSON**：
  - 以图搜人（`views/results.js`）：候选人结果只有 4 个统计数字 + JSON，
    没有候选人卡片——没有头像、姓名、相似度条。这是全系统最核心的业务场景，却是体验最差的页面。
  - 人像比对：只有「通过/未通过 + 4 个数字」摘要，没有左右图并排对照、没有分数可视化。
  - 总览页「平台状态」直接展示 Prometheus 原始指标树。
- `runtime/formatting.js` 维护了 **200+ 条 `fieldLabels`**；连同 `valueLabels`，字段/值翻译映射合计超过 300 条，
  是「直接渲染接口响应、未做产品化视图」的典型症状修补。

### 1.2 信息架构照抄后端 API，而非用户任务

- 导航 7 组 27 页，基本一个 API 端点一个页面：
  - 「人像比对」与「融合比对」是两个页面，但对用户是同一任务；
  - 「图片解析 / 视频任务 / 实时视频流 / 解析结果」把发起任务和查看结果拆散在 4 个页面；
  - 「特征重建」等管理员低频操作与日常操作平级；
  - 错误码、OpenAPI 定义等文档性内容占据一级导航。
- 业务操作员、接入开发者、平台管理员三种角色的功能混在同一层级。

### 1.3 工程术语直接暴露

- 下拉选项含 URL 路径（`人体解析 /v1/infer/persons`）；
- 表单默认暴露置信度/IoU/最大目标数/批次大小等算法参数，无「高级参数」折叠；
- 元数据要求用户手写 JSON；时间筛选要求输入 Unix 秒；
- 「阈值方案」是自由文本输入而非下拉；
- 危险操作靠手动输入确认词（`cleanup`/`backup`）或原生 `window.confirm`。

### 1.4 任务流靠「复制粘贴 ID + 按钮组」驱动

- 视频任务/视频流页面：一个 ID 输入框 + 查询状态/查看结果/取消/实时订阅 4 个按钮。
  **没有任务列表**，用户必须记住任务 ID；无自动轮询，WebSocket 需手动订阅。

### 1.5 反馈机制原始

- 全局唯一反馈渠道是侧栏底部一行 `status-strip` 文本，错误时直接展示
  `JSON.stringify` 后的原始报错；无 Toast、无骨架屏、无表单内联校验。

### 1.6 数据真实性混乱

- 应用凭证、Webhook 在服务端接口失败时**静默回退到 localStorage 演示数据**
  （`views/app.js`），调用日志失败时回退到「当前会话」本地记录，用户无法分辨真伪。
- 人员库「特征分布」散点图是假的（x 坐标 = `(personIndex*7 + featureIndex*3) % 12`），
  属于误导性可视化。

### 1.7 基础工程缺陷

- 无路由：当前视图存 localStorage，无 URL hash——不能深链、不能前进/后退、刷新丢上下文；
- 登录页同时要求理解租户标识/接口密钥/JWT 三个概念，无引导；
- `views/app.js` 中有一份与 `views/navigation.js` 完全重复的导航兜底数据（Unicode 转义形式），
  双份维护必然漂移；部分模板混有 `\uXXXX` 转义中文；
- 页面标题固定不变，tab 组件焦点管理不完整。



## 2. 重建决策：为什么重建而不是原地重构

### 2.1 结论

**重建，实施期采用绞杀者模式渐进替换，不推倒重来。** 迁移期旧版曾保留在 `/console/legacy`
作为对照与回退；截至 0.11.1 批次 E，旧版源码、旧路由和运行时灰度开关均已删除，回退只通过上一版镜像或受控静态构件完成。

### 2.2 依据

| 判断维度 | 本项目情况 |
|---|---|
| 现有架构能否支撑目标形态 | 不能。任务列表 + 实时进度、详情抽屉、hash 路由、Toast、键值对编辑器、按角色导航，每一项都需要组件化 + 响应式状态。在字符串模板 + 手动 DOM 绑定架构上实现等于徒手重造框架 |
| 可保留代码占比 | 低。产品化改造本来就要重写全部 27 个视图的渲染层（约 70% 代码量），原地重构与重建工作量接近，但产出可维护性差距巨大 |
| 重写风险是否有界 | 基本有界。现有行为是「表单 → fetch → 渲染」+ 2 条 WebSocket，可枚举、可对照验收；但批次 B 所需的任务列表、人员列表以及权限能力查询尚无完整契约，必须先补齐第 8.1 节接口 |
| 回退能力 | 迁移期依赖 `/console/legacy`；0.11.1 后不再存在运行时 legacy，必须部署上一版镜像或受控静态构件 |

### 2.3 不重建的唯一场景（已排除）

若控制台定位是「内部调试工具」且将来另有业务前端，则不值得重建。
但本控制台有租户登录、应用凭证管理、接入中心，README 亦称「业务控制台」——它就是产品本体。



## 3. 设计原则

以下原则贯穿所有批次，验收时逐条对照：

1. **默认给结论，按需给数据**。每个业务场景有专属结果组件；JSON 永远不是默认展示层，
   只出现在「开发者模式」开关打开后的详情抽屉「原始数据」标签页中。
2. **列表优先，ID 不需要记忆**。所有异步资源（视频任务、视频流、人员、应用、Webhook）
   一律以列表呈现，操作是行内按钮，详情用抽屉；创建后自动进入详情，绝不让用户复制粘贴 ID。
3. **按角色组织导航**：业务操作员（工作台）/ 接入开发者（开发者中心）/ 平台管理员（系统管理）。
4. **工程参数收进「高级参数」折叠区**，默认值可直接使用；URL 路径、算法参数、Unix 时间戳
   不出现在默认视图。
5. **状态自动更新**。运行中的任务/流自动接 WebSocket 或轮询，不需要用户手动点「刷新」或「订阅」。
6. **反馈即时且人话化**。操作用 Toast 反馈；错误码映射为中文说明并附「查看调用日志」链接。
7. **诚实的数据来源**。接口失败显示明确错误横幅，绝不静默回退本地演示数据；
   本地缓存数据必须带「本地缓存」标识。
8. **不做假可视化**。图表的每个坐标轴必须有真实含义，否则删除。
9. **危险操作分级确认**。普通单项删除使用统一确认组件；批量清理、非预演发布、特征重建等高影响操作追加影响摘要、二次确认或重新认证，替换 `window.confirm`。
10. **可深链**。每个页面（含抽屉中的资源详情）有稳定 URL，支持刷新保持与前进/后退。
11. **权限由服务端能力驱动**。导航、按钮和数据访问根据服务端返回的权限集合渲染；前端隐藏仅改善体验，不能替代后端鉴权。
12. **敏感数据最小驻留**。API Key/JWT、人员图像、原始响应和任务结果不得写入 `localStorage`；退出、租户切换和凭证失效时必须清理内存、会话缓存与对象 URL。

## 5. 技术选型

| 项 | 选择 | 理由 |
|---|---|---|
| 运行时与构建 | **Node.js 22 LTS + Vite** | 批次 A 同步升级仓库 CI；用 `.nvmrc`、CI 与 `engines` 固定同一主版本，产物 hash 化 |
| 框架 | **Vue 3（Composition API）+ TypeScript** | 模板语法对以 Python 为主的团队上手成本低；TS 承载生成后的 v1 API 契约 |
| 组件库 | **Element Plus** | 表格、抽屉、表单校验、消息与日期选择能力完整；是否满足严格 CSP 以批次 A 浏览器实测为准 |
| 状态 | **Pinia** | 只存 UI 状态、服务端能力和非持久任务视图；API Key/JWT 不进入持久化插件或 `localStorage` |
| 路由 | **Vue Router（hash 模式）** | 后端无需通配业务路由，`/console#/analysis/video/xxx` 可直接加载；鉴权后恢复目标路由 |
| 请求 | 原生 fetch 封装（移植现有 `apiRaw`） | 统一注入鉴权头、超时/取消、错误解析、`request_id` 提取和 401/403 处理 |
| API 契约 | **OpenAPI 自动生成 TypeScript 类型** | CI 重新生成并做差异检查，禁止只靠人工维护 `types.ts` |
| 图表 | **ECharts（按需引入，Canvas 优先）** | 仅总览指标与真实质量分布使用；引入前完成 CSP、包体和无障碍降级验证 |
| 测试 | **Vitest + Vue Test Utils + Playwright + axe** | 覆盖 API/组件、主链路、权限、CSP、可访问性和视觉回归 |
| 代码规范 | ESLint + Prettier + vue-tsc | 纳入根目录统一 CI 门禁 |

依赖版本在批次 A 的技术 ADR 中锁定，根目录提交唯一 `package-lock.json`，CI/Docker 一律使用 `npm ci`。
本方案确定采用 Vue 技术栈；实施期间若要改为 React，视为架构变更，必须先更新 ADR、估算和本文档，不能作为无成本替换项。

工程规范：

- **所有源码使用 UTF-8 明文中文**，禁止 `\uXXXX` 转义；
- 根 `package.json` 配置 npm workspace 并代理 `console:lint`、`console:test`、`console:typecheck`、`console:build`、`console:e2e`；console-next 包使用 ESM；
- 根目录只保留一个锁文件，依赖安装统一 `npm ci`，禁止 CI 隐式更新依赖；
- 单一导航数据源（路由 meta 驱动侧栏），路由 meta 声明所需权限，禁止兜底副本；
- 每个视图切换时更新 `document.title`（`影鉴 · <页面名>`），关闭抽屉时恢复触发点焦点和父路由；
- `npm run console:build` 产物输出到 `frontend/console-next/dist/`；Docker 的 Node 构建阶段生成产物，运行镜像只复制 `dist/`，不依赖开发机预构建目录；
- CI 顺序为 `npm ci` → OpenAPI 生成差异检查 → lint → unit/typecheck → build → Playwright；
- 生产构建关闭 source map 或将其作为受控构件保存，不通过静态路由公开；
- 所有列表使用服务端分页；图片缩略图懒加载并及时释放对象 URL，不在 Pinia/localStorage 缓存人员图像。

## 8. 后端配合改造

后端改动不只集中在静态路由。批次 A/B 必须完成以下接口、鉴权、静态服务和构建链路，
并同步更新 OpenAPI、租户隔离测试、权限测试与部署门禁。

### 8.1 批次 B 前置业务契约

| 接口 | 用途与最小响应 | 权限 | 验收要求 |
|---|---|---|---|
| `GET /v1/jobs` | 视频及其他异步任务列表；支持 `kind/status/created_since/created_until/cursor/limit`，返回 `items + pagination` | `jobs:read` | 当前租户隔离；稳定排序；运行中、完成、失败、已取消均可查询 |
| `GET /v1/gallery` | 人员分页列表；支持名称/ID 搜索、模态筛选、`cursor/limit`，返回不含 embedding 的人员摘要、特征数和受控缩略图引用 | `gallery:read` | 当前租户隔离；禁止返回向量；空库和游标失效有稳定响应 |
| GET /v1/console/me | 返回 tenant_id/auth_kind/roles/permissions/scopes/expires_at，作为导航、按钮与路由守卫的数据源；不再返回 legacy features | 任何已认证主体 | 不要求 admin:status；API Key、JWT、全局令牌分别有契约测试 |
| `POST /v1/console/ws-ticket` | 按请求的 job/stream 资源和权限签发一次性短期票据，供同源 WebSocket 握手 | 对应 `jobs:read` 或 `streams:read` | TTL 不超过 60 秒、一次使用、绑定租户与资源，日志只记录票据指纹 |

说明：`/v1/analysis/results` 只能覆盖已归档结果，不能替代运行中任务列表；前端不得通过枚举 ID、
读取进程内状态或拼接多个非分页接口模拟集合查询。新接口的分页结构复用现有 `pagination` 契约。

### 8.2 控制台静态壳与鉴权边界

1. `/` 是统一登录入口，认证成功后进入 `/console`；`/console/next` 仅作为新版直达验收别名。迁移期 `/console/legacy` 已在 0.11.1 删除，不再是运行时回退入口。所有静态壳均不含敏感数据，所有 `/v1/*` 接口继续执行认证、租户绑定和权限校验。
2. 未登录用户访问任意 `/console#<目标路由>` 深链时进入 `/?redirect=<目标路由>`；校验成功后恢复为 `/console#<目标路由>`。租户由单租户 API Key/JWT 自动解析，多租户身份应在后续身份系统中受控选择，不在登录页手工填写租户。
3. 迁移期 `/v1/console/me.features` 曾返回三个 v2 开关；0.11.1 后服务端不再返回已删除的 feature flag，导航和按钮只消费主体、租户、角色、权限、scope 与过期时间。
4. `/assets/console-next/{path}` 服务 `frontend/console-next/dist/`，复用 `relative_to` 路径白名单；使用标准 MIME 推断，拒绝目录、`index.html` 和 source map 越权读取。
5. 带 hash 的 JS/CSS/字体/图片使用 `Cache-Control: public, max-age=31536000, immutable`；HTML 使用 `no-cache`，敏感 API 和鉴权对象响应使用 `Cache-Control: no-store`。
6. 旧版与新版静态壳可公开不代表数据接口公开；自动化测试必须验证匿名请求只能取得壳和静态资产，不能取得任何租户数据。

### 8.3 CSP 与前端依赖验证

- Vite 外链产物允许使用 `script-src 'self'`，不假定现有静态 HTML 存在 nonce 注入；如未来存在必要的内联引导脚本，必须逐请求注入真实 nonce 并有响应测试。
- 批次 A 用锁定版本实测 Element Plus、Popper、ECharts 和 Vue 过渡效果产生的所有 CSP 违规。优先通过 CSS 类、关闭非必要动画或替换组件维持严格策略。
- 如果运行时定位确实依赖动态 `style` 属性，只能在 ADR 和安全评审通过后，使用 CSP Level 3 将例外收窄到 `style-src-attr`；`script-src` 永不加入 `'unsafe-inline'` 或 `'unsafe-eval'`。
- Playwright 在真实响应头下收集 `securitypolicyviolation` 和控制台错误；存在未登记违规时批次 A/B 不得验收。

### 8.4 凭证与敏感数据

- API Key/JWT 默认只保存在当前标签页的 `sessionStorage`，不进入 Pinia 持久化、`localStorage`、URL、错误上报或分析事件；关闭标签页即失效。首版不提供“记住我”。
- 退出、401、凭证过期、租户切换时清理凭证、能力状态、任务缓存、对象 URL 和剪贴板提示状态；路由回到登录页。
- WebSocket URL 只携带一次性 ticket，不携带主 API Key/JWT；网关、应用和审计日志对 ticket、token、authorization、缩略图 data URL 做统一脱敏。
- `RawDataDrawer` 经过递归脱敏后才能显示/复制，默认屏蔽密钥、token、embedding、生物特征向量、对象存储内部地址和服务端栈信息。
- 浏览器缓存只保存非敏感偏好；人员图像、任务结果和原始响应不离线持久化。

### 8.5 构建、部署与仓库门禁

- 根 `package.json` 升级为 npm workspace，保留 Node SDK 测试并增加 console-next 代理脚本；提交唯一锁文件。
- Dockerfile 增加独立 Node 构建阶段：`npm ci` 后构建 console-next，运行镜像只复制 `dist/`，不复制 `node_modules`，也不依赖开发机生成物。
- CI 固定 Node.js 22，执行 OpenAPI 类型生成差异检查、lint、单测、typecheck、build 和 Playwright；构建后的入口及 manifest 必须存在。
- 同步修改 `tools/deploy_check.py`、`tools/readiness/structure.py`、`tools/readiness/sources.py`、控制台截图验收工具和相关契约测试。删除旧版前，门禁同时识别 legacy 与 next；批次 E 再删除 legacy 条目。
- README、部署文档、镜像构建缓存、依赖许可证/漏洞报告和前端 SBOM 纳入交付物。



## 9. 分批迁移计划（按角色与路由渐进替换）

每批独立部署、按能力开关灰度，不在批次 B 对所有角色整体翻转 `/console`。批次内页面达到第 10、11、12 节规范，
并满足本节退出条件后才算完成。以下日历估算基于 **2 名前端 + 0.5 名后端 + 1 名兼职 QA/产品验收**；人员配置变化后重新估算。

### 批次 0：契约与设计冻结（约 2-3 天）

- 确认 Vue 技术 ADR、Node/依赖版本、CSP 策略和浏览器范围；
- 冻结旧版行为对照表，补齐页面 → 接口 → 权限 → 状态 → 测试映射；
- 评审 `GET /v1/jobs`、`GET /v1/gallery`、`GET /v1/console/me`、WS ticket OpenAPI；
- 为工作台关键页面完成低保真流程和数据密度评审。

**退出条件**：接口 schema 与权限矩阵获后端/安全/前端共同确认；未决项有负责人和截止日期。

### 批次 A：骨架、安全与交付链路（约 4-5 天）

- 初始化 Vite + Vue 3 + TS + Element Plus + Pinia + Router、npm workspace 和锁文件；
- 完成 `api/client.ts`、`api/ws.ts`、OpenAPI 类型生成、请求取消与统一错误模型；
- 后端提供公开静态壳、`/v1/console/me`、WS ticket、`/console/next` 与 `/assets/console-next/`；
- 登录页、路由回跳、权限导航、会话清理和 401/403 流程；
- `ConsoleLayout`、`RawDataDrawer`、`EmptyState`、分级 `DangerConfirm`；
- Docker/CI/readiness 双版本门禁；在真实 CSP 下完成组件库与 ECharts 替代方案验证。

**退出条件**：匿名只能加载静态壳；三种凭证能登录并获得正确能力；刷新/深链回跳正确；凭证不进入 localStorage/URL；
401/403/500 可理解；CSP 零未登记违规；CI 与容器构建全部通过。

### 批次 B：工作台（约 2-3 周，价值最高）

- 先交付并验证任务列表、人员列表两个后端集合接口；
- 总览（状态卡片 + 指标，含 P99/队列深度）；
- 智能分析（三 tab：图片/视频/视频流；任务列表化 + 自动实时更新 + 详情抽屉）；
- 比对（合并融合；判定横幅 + 并排对照 + 相似度条）；
- 以图搜人（候选人卡片列表）；
- 人员库（表格 + 详情抽屉 + 注册向导 + 高级操作里的特征重建）；
- Playwright、axe、视觉回归、WebSocket 重连/轮询降级和大列表/图片性能验收。

**灰度方式**：只为工作台能力开启 `console_workbench_v2`，开发者中心和系统管理入口继续明确跳转 legacy 对应页；
先内部租户，再 10%、50%、100% 工作台用户。每级至少观察一个工作日且无 P0/P1 问题再扩大。

### 批次 C：开发者中心（约 1-1.5 周）

- 接入配置（应用凭证 + Webhook 合并，服务端数据为唯一来源，失败显示错误横幅）；
- 调试台（调试 / SDK 示例 / 错误码 / OpenAPI 四 tab）；
- 调用日志（本地日期时间筛选、应用下拉、错误码说明）；
- 为具备对应权限的主体开启 `console_developer_v2`，未迁移能力继续跳 legacy。

### 批次 D：系统管理（约 1-1.5 周）

- 模型中心（模型 / 发布 / 评估；发布默认预演，非预演使用高风险确认）；
- 阈值与标注（阈值下拉编辑 + 轨迹审阅标注流）；
- 运维与合规（SLO / 告警 / 备份保留 / 审计；清理与备份使用高风险确认）；
- 为管理员/审计员按权限开启 `console_admin_v2`，完成全角色迁移。

### 批次 E：观察与清理（至少 2 个发布周期，清理工作约 2-3 天）

- 全角色 100% 使用新版后继续保留 `/console/legacy`，至少观察 14 天；
- 旧版访问占比低于 1%，无未解决 P0/P1、无因新版导致的权限/租户隔离事故，回退演练通过；
- 删除 `frontend/console/`、`/console/legacy` 和全部 legacy feature flag；
- 更新根 npm 脚本、部署/就绪/截图工具、契约测试、README 与部署文档；
- 清理无引用字典和双版本兼容代码，归档旧版行为对照表与最终验收报告。

**估算**：从批次 0 到全角色功能完成约 5-7 周，之后至少 14 天观察期；估算不包含新增接口评审延误、设计返工和生产问题修复。

### 9.1 灰度观察与自动回退指标

每个灰度级别比较新旧版同租户基线，至少监控：登录成功率、前端未处理异常率、API 4xx/5xx、主流程成功率、
WebSocket 重连次数、页面加载 P75、用户主动切回 legacy 比例。满足任一条件立即停止扩大并关闭对应 feature flag：

- 出现权限绕过、跨租户数据、凭证泄漏或不可逆错误操作；
- 新版主流程成功率相对旧版下降超过 2 个百分点；
- 前端未处理异常影响超过 1% 会话，或 P75 加载时间连续 30 分钟劣化超过 30%；
- 同一批次出现未解决 P0，或影响主流程的 P1。

## 11. 全局体验与安全规范

1. **反馈分层**：提交、保存、取消、删除等状态变更操作使用 Message/Notification；字段错误就地显示，页面级失败使用错误横幅，避免所有结果都用 Toast。顶栏只在存在实时连接时显示细连接状态。
2. **错误人话化**：`client.ts` 解析 `error_code` 并映射中文说明；未知错误显示 `request_id`。仅当用户具备调用日志权限时显示「查看调用日志」链接，禁止展示服务端栈和内部地址。
3. **加载与取消**：首屏表格/卡片使用稳定尺寸骨架；路由切换、重复提交和上传取消时终止无效请求，禁止旧响应覆盖新筛选条件。
4. **空状态**：使用 `EmptyState`，文案说明当前结果并提供一个最相关操作；无权限与无数据是不同状态，不能都显示“暂无数据”。
5. **开发者模式**：开关只持久化为非敏感偏好。具有相应权限时才显示 `RawDataDrawer`，内容先按第 8.4 节递归脱敏；关闭时不渲染 JSON 容器。
6. **登录与会话**：默认单输入框（接口密钥）+「高级选项」（租户标识、JWT）；凭证只保存当前标签页，显示会话过期时间，不提供首版“记住我”。401、过期和租户切换执行统一清理。
7. **深链与历史**：列表筛选、分页、tab 和详情资源 ID 写入路由；抽屉关闭、浏览器前进/后退、刷新和登录回跳得到一致状态。不得把凭证、图像或完整查询元数据放入 URL。
8. **可访问性**：路由切换更新标题和主内容焦点；弹窗/抽屉焦点圈定并恢复；tab、菜单和表格操作支持键盘；动态状态使用 `aria-live`；颜色不是唯一状态信号；支持 `prefers-reduced-motion`。自动化 axe 不得有 serious/critical 问题，关键流程还需键盘人工验收。
9. **响应式范围**：主要生产工作区按 1280px 及以上优化；768-1279px 保持完整能力并允许侧栏折叠；360-767px 支持登录、状态查看、搜索和关键审批，宽表采用列优先级/横向滚动，不把完整桌面表格强塞成卡片嵌套。
10. **浏览器范围**：支持 Chrome/Edge 最近两个主版本、当前 Firefox ESR 和当前 Safari 主版本；批次 0 锁定具体测试版本，低于范围显示明确的不支持提示。
11. **时间与数字**：API 保持 UTC/ISO 8601；界面按用户本地时区展示并在筛选区标明时区，复制详情时同时提供原始 UTC。相似度、百分比、字节和持续时间使用统一格式化函数。
12. **性能与隐私**：路由级代码分割；ECharts、管理页和开发者工具延迟加载；列表服务端分页；缩略图懒加载。退出或离开详情时释放 Blob URL，页面隐藏时暂停非必要轮询和动画。

### 12.3 测试分层

- **后端契约**：pytest 覆盖新增列表、`/v1/console/me`、WS ticket 的正常/分页/游标失效/401/403/跨租户/脱敏；验证静态壳匿名可取而租户数据不可取。

- **前端单元**：API 请求构造、超时/取消、错误模型、权限判断、格式化和脱敏；`VerdictBanner`、`SimilarityBar`、`KVEditor`、分级确认组件及路由守卫。
- **组件集成**：表格分页/筛选竞态、抽屉路由同步、上传取消、401 清理、WS 重连后去重、轮询降级和空/错/禁用四态。

- **测试数据**：使用确定性、租户隔离的 API fixture 或专用测试环境；禁止通过生产代码静默回退 localStorage 假数据。涉及生物特征的 fixture 必须为合成或经批准数据。

### 12.4 非功能预算

- 首个业务路由初始 JS gzip 不超过 300 KB、CSS gzip 不超过 100 KB；ECharts、开发者中心和系统管理必须独立懒加载。超预算需 ADR 说明和批准。
- 目标网络/设备基线由批次 0 固化；在该基线上主路由 LCP P75 ≤ 2.5 秒、INP P75 ≤ 200 ms、CLS ≤ 0.1。
- 单页 DOM 不同时渲染超过 200 条业务行；默认分页不超过 50 条，最大分页由后端上限约束。
- 运行中列表在前台的状态延迟目标 ≤ 3 秒；WebSocket 不可用时轮询退避上限 30 秒，并明确显示“实时连接已降级”。
- axe serious/critical 为 0；关键文本、控件和状态颜色达到 WCAG 2.1 AA，并保留实际测量记录，不能只依据旧 token 推断。



## 13. 风险与回退

| 风险 | 缓解与责任边界 |
|---|---|
| 任务/人员列表等契约缺失 | 批次 0 先冻结第 8.1 节 OpenAPI；接口未通过租户/权限/分页测试，批次 B 不开始对应页面 |
| 静态壳受保护导致无法登录 | 批次 A 明确壳与数据 API 的鉴权边界，并做生产配置下的匿名壳/受保护数据测试 |
| 凭证或生物数据在浏览器持久化 | sessionStorage 最小驻留、统一清理、RawData 脱敏、WS ticket 与自动化泄漏断言 |
| CSP 与组件库/图表冲突 | 批次 A 在锁定依赖和真实 CSP 下验证；必要例外必须 ADR + 安全批准，绝不放宽脚本策略 |
| 重写遗漏旧版行为 | 第 12.2 节对照表作为 PR 门禁；0.11.1 删除 legacy 后，回归保护改由对照清单、契约测试、Playwright 和 readiness 断言承担 |
| API 在重建期间演进 | OpenAPI 自动生成并由 CI 做差异检查；破坏性变更走版本化或兼容期，不能由本地 typecheck 代替 |
| 批次 B 影响未迁移角色 | 迁移期通过工作台/开发者/管理三个独立 feature flag 控制；0.11.1 后该风险关闭，不再允许跳转 legacy |
| Docker、部署和就绪工具遗漏旧路径 | Docker/CI/deploy/readiness/截图/契约测试已在批次 E 切到 Console Next，并新增旧路径回归断言 |
| 团队 Vue 经验或估算不足 | 批次 0 做技术 spike 和故事拆分；以实际吞吐重估，范围优先级高于固定日期 |
| 高风险操作误触 | 默认预演、影响摘要、权限复核、二次确认/重新认证、审计 request_id；回退不能替代操作前保护 |

### 13.1 回退机制

1. 0.11.1 后不再存在运行时 legacy 路由或 v2 feature flag；`/console/legacy` 必须稳定返回 404。
2. 每次发布保留可寻址的上一版镜像摘要或经过签名/校验的静态构件；紧急回退通过部署上一版镜像或受控静态构件完成，不通过环境变量切回旧路由。
3. 回退不得删除业务数据、解析档案、对象存储载荷或审计记录；只允许替换控制台静态壳和服务镜像。
4. 命中第 9.1 节历史灰度条件对应的安全、跨租户、凭证或 CSP 回归信号时，发布负责人停止放量并执行上一版镜像/构件回退，按安全事件流程处理。
5. 回退后保留版本、触发指标、request_id、审计记录和构件校验摘要；问题修复后重新从内部租户开始验证。

### 13.2 删除旧版的最终门槛

批次 E 已在 0.11.1 执行：`frontend/console`、`/console/legacy`、旧 `/assets/console*` 和 legacy feature flag 已删除。
后续门槛转为回归保护：README/部署/就绪工具必须继续指向 Console Next；测试和 readiness 必须阻断旧目录、旧静态路径、旧 DOM 标记和旧灰度变量回归。
