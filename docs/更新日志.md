## [0.0.1] - 2026-06-03

首次建立 `gpu-services` 更新日志。由于此前没有独立更新日志文档，本版本将当前系统已经具备的全部功能统一记录为 `0.0.1` 的首次版本内容。

说明：本文档中的版本号是推理服务项目版本。`models.yml` 或模型卡中的 `version` 是算法模型包版本，二者相互独立。

### 服务定位

- 面向 Ubuntu、Docker、NVIDIA GPU 的共享 ONNX 推理服务。
- 使用 FastAPI 提供 HTTP API。
- 面向图像识别、人体检测、ReID、人像检索、多帧分析、离线视频抽帧推理、视频流抽帧推理等业务场景。
- 定位为算法落地与推理服务，不承载训练、标注和模型研发平台本体。
- 与训练/标注/研发侧通过标准模型包、模型卡、labels、sha256、样例和验收报告对接。

### 架构与模块化

- `main.py` 保留为兼容入口，仅导入 `app.server:app`。
- `app/server.py` 负责 FastAPI 应用装配、middleware 和 startup。
- `app/routes.py` 作为总路由聚合器。
- 路由已拆分为健康检查、模型查询、模型生命周期、原始 tensor 推理、通用视觉、人像业务、灰度治理和调试接口。
- 运行时已拆分为状态、session 创建、模型注册缓存和推理执行模块。
- 模型配置已拆分为加载、状态、访问、引用解析和写入模块。
- 图像与视频处理已拆分为图片 IO、视频 IO、预处理、几何/NMS 和后处理模块。
- 推理编排已按检测、ReID、分类、组合轨迹拆分。
- `core.py`、`runtime.py`、`vision.py`、`inference.py`、`model_config.py` 保留为兼容导出层。

### 通用 ONNX 推理

- 保留 `/predict` 作为 raw tensor 通用推理接口。
- 支持 JSON tensor 输入。
- 自动识别 ONNX 输入 dtype。
- 支持 `float32`、`float16`、`float64`、`int32`、`int64`、`bool` 等常见 dtype 映射。
- 返回全部 ONNX 输出。
- 返回 `input_shape`、`output_shapes`、冷加载状态、排队耗时、加载耗时、推理耗时和总耗时。
- 使用 `MAX_TENSOR_ITEMS` 限制输入 tensor 元素数量。

### 通用视觉推理

- 新增并提供 `POST /vision/infer`。
- 新增并提供 `POST /vision/batch-infer`。
- 支持通过 `model_id` 调用模型别名。
- 支持通过 `project_name` 和 `model_name` 直接调用模型。
- 支持检测、分类、ReID 三类任务自动分发。
- 支持上传单张或多张图片。
- 支持 `MAX_VISION_IMAGES` 限制单次图片数量。
- 检测任务支持 confidence、iou、max_detections、class_filter、labels 和 NMS。
- 分类任务支持 top_k、softmax、sigmoid、raw/identity、threshold 和 labels。
- ReID 任务支持 ImageNet normalize、L2 embedding normalize 和 `include_vectors`。
- 支持 `traffic_key`，用于 weighted rollout 稳定分流。

### 人体检测与 ReID 业务接口

- 提供 `POST /infer/persons`。
  - 支持图片或多帧图片上传。
  - 服务内完成解码、letterbox、YOLO 推理、person 类过滤和 NMS。
  - 返回人体框、置信度、类别 ID 和类别名。
- 提供 `POST /infer/person-embeddings`。
  - 支持人体 crop 图片上传。
  - 服务端完成 ReID 预处理和 embedding 推理。
  - 支持是否返回完整 embedding 向量。
- 提供 `POST /infer/person-tracks`。
  - 组合检测和 ReID。
  - 先检测人体，再裁剪人体并生成 embedding。
  - 返回每帧 persons、embedding_index、embedding_dim，可选返回 embedding。
  - 不伪造跨帧 `track_id`，轨迹关联由调用方或后续业务模块完成。

### 离线视频与视频流推理

- 提供 `POST /infer/video/person-tracks`。
  - 支持上传离线视频文件。
  - 使用 OpenCV 抽帧。
  - 支持 `frame_interval` 和 `max_frames`。
  - 返回 `source_frame_index` 和可推导的 `source_seconds`。
- 提供 `POST /infer/stream/person-tracks`。
  - 支持 RTSP、RTMP、HTTP、HTTPS 视频流。
  - 默认关闭服务端主动拉流能力。
  - 设置 `ALLOW_STREAM_URLS=true` 后启用。
  - 支持 `frame_interval`、`max_frames`、`read_timeout_seconds`。

### 模型配置

- 使用 `models.yml` 管理模型和别名。
- 支持 `aliases` 和 `models`。
- 支持 `task`、`type`、`runtime`、`version`、`precision`、`input`、`output`、`artifact`、`rollout`、`max_concurrency`、`queue_timeout_seconds` 等字段。
- 兼容旧字段 `input_size`、`confidence`、`iou`、`classes`、`normalize`、`embedding_normalize`。
- 支持 `/reload-config` 在不重启服务的情况下重新读取配置。
- 支持 active、candidate、staging 等模型上线状态描述。

### 标准模型包

- 支持模型文件按 `project/model.onnx` 放置在共享目录。
- 支持模型卡 `<model>.model-card.yml` 或 `artifact.model_card` 指定路径。
- 支持 labels 文件 `<model>.labels.txt` 或 `artifact.labels` 指定路径。
- 支持 `output.classes: coco`。
- 支持 inline classes。
- 支持 `artifact.sha256`。
- 模型加载时可校验 sha256。
- `/model-package` 可返回模型包、模型卡、labels 和 sha256 匹配状态。
- 提供 `tools/validate_model_package.py` 做模型包静态校验。

### 模型加载、缓存与生命周期

- 模型按 `project_name/model_name` 懒加载。
- 模型缓存为单 worker 本地缓存。
- 首次并发加载同一模型时使用加载锁，避免重复加载。
- 支持 `MAX_LOADED_MODELS`。
- `MAX_LOADED_MODELS=0` 表示不限制缓存数量。
- 正整数启用 LRU 淘汰。
- 支持 `WARMUP_MODELS` 启动预热。
- 支持 `POST /warmup` 手动预热。
- 支持 `POST /unload` 手动卸载。
- 支持 `POST /reload` 手动重载。
- `/models` 返回已加载模型、缓存数量和模型详情。
- `/model-info` 返回模型输入输出、provider、hash、文件大小、加载时间和推理次数。

### GPU 队列、并发与性能治理

- `GPU_QUEUE_LIMIT` 控制单 worker 同时进入 GPU 推理段的请求数量。
- `MODEL_CONCURRENCY_LIMIT` 控制单模型默认并发。
- 模型配置可通过 `max_concurrency` 或 `runtime.max_concurrency` 覆盖单模型并发。
- `MODEL_QUEUE_TIMEOUT_SECONDS` 控制模型队列和 GPU 队列等待超时。
- 模型配置可通过 `queue_timeout_seconds` 或 `runtime.queue_timeout_seconds` 覆盖队列超时。
- 队列等待超过配置值会返回 503，避免请求无限堆积。
- 支持单请求内 batch。
- 如果模型不支持动态 batch，检测/ReID 等路径可自动降级为逐帧推理。
- FP16 ONNX 输入会根据 session 输入 dtype 自动 cast。

### TensorRT Execution Provider

- 支持模型配置 `runtime: tensorrt`。
- 需要设置 `ENABLE_TENSORRT=true`。
- 运行环境必须实际提供 ONNX Runtime `TensorrtExecutionProvider`。
- 支持 TensorRT engine cache 配置：
  - `TENSORRT_ENGINE_CACHE_ENABLE`
  - `TENSORRT_ENGINE_CACHE_PATH`
- TensorRT provider 不可用时，模型加载会给出明确错误。

### 灰度、别名、回滚与审计

### 部署

- 使用 Docker Compose 启动两个 GPU worker。
- 默认 worker 为 `gpu-worker-0` 和 `gpu-worker-1`。
- 默认宿主机端口为 `127.0.0.1:9001` 和 `127.0.0.1:9002`。
- 默认容器服务端口为 `8000`。
- 每个 worker 可绑定不同 GPU：
  - `GPU_WORKER_0_DEVICE`
  - `GPU_WORKER_1_DEVICE`
- 容器内模型目录为 `/models`。
- 共享模型宿主机目录由 `MODELS_HOST_DIR` 配置。
- Dockerfile 基于 CUDA 11.8 runtime。
- 固定 `onnxruntime-gpu==1.18.0`。
- 设置 `TZ=Asia/Shanghai`。
- 提供 Ubuntu 部署文档 `DEPLOY_UBUNTU.md`。
