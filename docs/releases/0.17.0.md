# 影鉴 0.17.0 发布说明

- 发布日期：2026-07-22
- 发布类型：平台能力版本
- API 兼容性：保留现有 `/v1` 路径，统一响应增加 `schema_version: "1.0"` 并强化 OpenAPI 模型
- 数据兼容性：默认项目兼容既有租户数据；生产 PostgreSQL 必须应用新增表结构
- 部署兼容性：需要同步升级服务、Console Next 和 SDK，并重新执行平台与模型就绪门禁

## 版本摘要

0.17.0 围绕“强结构化契约、项目级隔离、完整生产能力”完成平台补齐。影鉴现在可以在一个租户内承载多个业务项目，并把项目边界贯穿凭证、请求上下文、业务数据、WebSocket、Webhook、调用日志和统计。公开响应具有版本化、可机器校验的统一外层，核心解析数据不再只依赖宽泛字典描述。

生产部署新增 PostgreSQL 访问状态 CAS、跨副本原子日配额和持久化调用日志。七类模型的生产配置模板已完整提供，但仓库不附带真实权重；完整生产就绪仍以部署方提供合法模型文件、真实 SHA256 并通过严格门禁为准。

## 强结构化契约

所有 `/v1` 路由使用共享契约路由器。成功响应稳定为：

```json
{
  "status": "success",
  "schema_version": "1.0",
  "request_id": "req-...",
  "data": {},
  "warnings": null,
  "meta": null
}
```

错误响应稳定为：

```json
{
  "status": "error",
  "schema_version": "1.0",
  "request_id": "req-...",
  "error": {
    "code": "stable_error_code",
    "message": "可读错误信息",
    "details": null
  }
}
```

- 人脸、人体、姿态、外观、步态、通用视觉和人员轨迹接口具有专用 OpenAPI 响应模型。
- 坐标、置信度、计数、特征维度、模型状态和耗时字段具备明确类型与范围约束。
- `include_vectors` 或对应向量开关仍由调用方控制；未请求向量时字段为 `null`，不会因执行路径不同而随机缺失。
- 契约顶层拒绝未知字段；允许演进的明细对象保留扩展能力，以便在不破坏旧客户端的前提下增加模型属性。
- Console Next 的 TypeScript OpenAPI 定义已经重新生成。接入方使用代码生成客户端时应重新生成并发布自己的客户端。

## 项目级隔离

### 项目目录

新增以下管理接口：

- `GET /v1/access/projects`
- `POST /v1/access/projects`
- `PATCH /v1/access/projects/{project_id}`

每个新租户自动拥有 `default` 项目。未显式选择项目的既有调用继续使用该项目；项目 ID 必须满足服务端格式约束，停用项目会立即拒绝该项目的新请求。

### 请求与凭证

- 非默认项目通过 `X-Project-ID` 选择。
- API Key 与接入应用的项目绑定，服务端会自动推导项目；显式发送其他项目会返回 `403`。
- JWT/OIDC 支持 `project_id`、`project` 和 `projects` claim；多项目凭证必须显式选择获授权的项目。
- 浏览器会话保存当前项目；Console Next 的 API 和 WebSocket 请求使用同一项目上下文。
- WebSocket ticket 同时绑定租户、项目、资源和权限，不能跨项目复用。

### 数据范围

以下业务资源使用 `tenant_id + project_id` 组合隔离：

- 人员库、特征和搜索；
- 图片解析、比对和人员轨迹；
- 视频任务、实时流和流事件；
- 解析档案与对象读取；
- 复核标注和相关统计；
- 接入应用、Webhook、调用日志和应用用量。

内部兼容命名空间只用于存储层，公开响应会分别返回真实 `tenant_id` 和 `project_id`，不会暴露内部组合值。

## PostgreSQL 生产能力

`tools/portrait_postgres_schema.sql` 新增：

| 表 | 用途 |
| --- | --- |
| `portrait_access_state` | 以 revision CAS 持久化租户、项目、成员、应用和 Webhook 状态 |
| `portrait_application_daily_usage` | 跨进程原子实施应用日配额 |
| `portrait_call_logs` | 持久化项目级调用日志、筛选、摘要和应用统计 |

访问状态写入带预期 revision；并发修改不会静默覆盖。日配额使用数据库条件更新，多个 API worker 共享计数。调用日志支持租户、项目、应用、request_id、接口、状态、错误码和时间窗口过滤，并提供持久化摘要。

PostgreSQL 连接池现在显式打开，覆盖多线程并发获取连接和关闭生命周期，兼容当前 `psycopg-pool` 行为。

`examples/production-models.example.yml` 包含以下生产候选配置：

| 能力 | 示例模型 |
| --- | --- |
| 人体检测 | YOLO |
| 人体 ReID | OSNet IBN |
| 外观属性 | Attribute ReID |
| 人脸检测 | SCRFD |
| 人脸特征 | ArcFace |

模板内所有 `replace-with-real-sha256` 都必须替换。仓库不保证示例文件名对应的权重授权、精度或业务阈值；部署方需要基于真实数据完成模型卡、SHA256、加载、精度、延迟、显存和阈值验收。

## 升级步骤

1. 进入维护窗口，停止权限目录、项目、应用、Webhook 和模型配置写入。
2. 备份当前提交 SHA、镜像、`.env`、`runtime-state`、PostgreSQL、对象存储和 Redis 未完成任务。
3. 拉取 `0.17.0`，安装生产可选依赖并检查配置：

```bash
git fetch origin
git pull --ff-only origin main
grep -F 'version = "0.17.0"' pyproject.toml
docker compose -p portrait-hub config --quiet
```

4. 对目标 PostgreSQL 应用幂等 schema：

```bash
export POSTGRES_DSN='替换为真实 PostgreSQL DSN'
psql "$POSTGRES_DSN" -v ON_ERROR_STOP=1 \
  -f tools/portrait_postgres_schema.sql
```

5. 生产环境设置 `PORTRAIT_STORAGE_BACKEND=postgres` 和真实 `POSTGRES_DSN`。镜像构建必须包含生产可选依赖（Compose 构建参数 `INSTALL_PROD_OPTIONAL=true`）。
6. 用真实权重初始化或合并 `runtime-state/models.yml`，不要用仓库模板覆盖已有运行配置；替换所有 SHA256 占位值。
7. 构建并滚动发布服务和控制台：

```bash
docker compose -p portrait-hub build --pull
docker compose -p portrait-hub up -d --remove-orphans
```

8. 验证默认项目能读取旧数据，创建第二个项目并确认凭证、人员库、任务、流、档案和日志不能跨项目访问。
9. 执行平台严格门禁；真实模型全部配置后再执行完整严格门禁。

新增表使用 `CREATE TABLE IF NOT EXISTS`，可以重复执行 schema 脚本。脚本不会删除旧表或业务数据，但正式执行前仍必须完成数据库备份和恢复演练。

## 接入兼容性

- 路径和 HTTP 方法保持 `/v1` 兼容；旧客户端未发送项目时进入 `default` 项目。
- 成功与错误响应新增 `schema_version`，核心接口 OpenAPI 更严格。只读取既有字段的客户端可以继续运行；按字段数量、未知字段或宽泛旧 schema 做严格反序列化的客户端必须升级。
- 新 API Key 绑定项目。旧应用数据会补齐到默认项目；迁移到非默认项目时应创建独立应用和密钥，不应复用默认项目高权限凭证。
- Python、Node、Go、Java SDK 版本均为 `0.17.0`，分别支持 `project_id`、`projectId`、`WithProjectID` 和 Builder `projectId`。

## 回退边界

回退到 `0.16.0` 时保留新增 PostgreSQL 表，不要删除它们；旧服务会忽略这些表。需要注意：

- 0.16.0 不理解非默认项目，升级后写入的多项目数据不能由旧版本安全继续处理。
- 一旦非默认项目产生业务数据，禁止在 0.16.0 上恢复写流量；只能用于只读排障或恢复到升级前数据库/运行状态快照。
- 回退服务、Console Next 和 SDK 应使用同一版本镜像/构件，避免新客户端发送旧服务无法识别的项目语义。
- 数据库恢复会覆盖升级后数据，必须由变更负责人确认恢复点与数据丢失范围。
