## [0.5.52] - 2026-06-30

本版本收口运行期状态持久化和长驻 stream daemon 的并发保护，降低高频写入、重复拉流和生产后端静默降级带来的风险，同时补齐配置与文档说明。

### 运行期持久化与状态安全
- 本地 JSON 底库 WAL 新增 `upsert_feature` 增量条目，单个特征写入不再反复落整个人员特征列表；WAL 回放会按 `feature_id` 去重并恢复人员更新时间。
- 视频任务运行中进度持久化改为轻量结果，仅保存 metadata、帧数量和 partial 状态，最终完成时再保存完整帧结果，减少长视频处理中间态 JSON 体积。
- PostgreSQL 底库快照替换改为按 snapshot 中涉及的 tenant 清理人员和特征，避免多租户部署中一次租户级恢复误删其它租户数据。

### Stream daemon 并发保护
- `StreamRecord` 增加 `worker_lease_owner` 与 `worker_lease_expires_at`，stream worker 会话启动前会先抢占可过期 lease，停止/重启流时会清理 lease 状态。
- `app.portrait_stream_worker_daemon` 新增每个 stream 的进程级原子 lock 文件兜底，使用 `O_CREAT | O_EXCL` 创建并写入 owner/token，避免多个 daemon 进程同时拉取同一条流。
- stale lock 可按 `STREAM_WORKER_PROCESS_LOCK_STALE_SECONDS` 回收；daemon 结束、异常或 lease 获取失败时会释放 state lease 和进程 lock。
- 新增配置 `STREAM_WORKER_LEASE_TTL_SECONDS`、`STREAM_WORKER_PROCESS_LOCK_STALE_SECONDS` 与 `STREAM_WORKER_LOCK_DIR`，并同步 `.env.example` 和本地 `.env` 配置项。

### 运行时指标与后端行为
- GPU 队列深度指标改用显式等待者计数，覆盖全局队列和按设备队列，避免依赖 `asyncio.Semaphore` 私有 `_waiters` 结构导致指标不稳定。
- pgvector/Qdrant 查询失败时会遵守 `PORTRAIT_REQUIRE_PRODUCTION_VECTOR_BACKEND`：生产强制后端模式下失败关闭，开发默认仍可回退本地扫描。
- Qdrant collection 初始化只在明确不存在时创建，非 404 类连接、权限或服务端错误会向上抛出，避免掩盖生产后端异常。

### 文档与配置
- README 补充 stream daemon lease/lock 行为、GPU 队列深度指标和新增环境变量说明。
- Ubuntu 部署文档补充 stream worker lock 目录、lease/stale lock 配置和长驻流 worker 运维注意事项。
- `.env` 已与 `.env.example` 的配置 key 和顺序对齐；密钥类本地值未改动。

### 测试与验证

## [0.5.47] - 2026-06-26

本版本推进模型生命周期“流程收口”。灰度（rollout）、回滚、版本别名此前已较完整（5 个 `/rollout/aliases/*` 端点 + 加权分流 + 乐观锁 + 审计 + 回滚到 previous_target，并有充分测试），本版本只补两个真正缺失的健壮性环节：预热容错与加载失败自愈。

### 预热容错（app/server.py）
- `warmup_models` 改为逐模型隔离：默认尽力而为，单个模型加载失败只记录脱敏摘要并继续预热其余模型，最后输出 `succeeded/failed/total` 汇总。此前任一模型加载失败会让异常透传至 lifespan，导致整个进程启动失败（其余可用模型也无法对外服务）。
- 新增 `WARMUP_FAIL_FAST`（默认 `false`）：严格部署可设为 `true`，恢复“任一预热失败即启动失败”的旧语义。

### 加载失败自愈（app/runtime_registry.py）
- 主模型注册表 `get_or_load_model` 新增加载失败冷却：某模型加载失败后写入“重试时间戳”，冷却窗口内的后续请求直接 `503` 快速失败，不再每次重做昂贵的 `model_hash` + `create_session`；窗口过后自动重试，实现加载失败自愈。加载成功立即清除冷却标记。
- 行为对齐既有的能力运行时冷却（`RUNTIME_CAPABILITY_RETRY_COOLDOWN_SECONDS`），新增独立的 `MODEL_LOAD_RETRY_COOLDOWN_SECONDS`（默认 30s，`<=0` 关闭）。
- 新增指标 `gpu_worker_model_load_cooldown_rejections_total`：冷却窗口内被快速拒绝的请求数。

### 测试
- `tests/test_model_package_runtime.py`：加载失败设置冷却并使后续请求 `503` 快速失败（不重试加载）；冷却过期后允许重试、成功后清除冷却标记。
- `tests/test_server_warmup.py`：默认预热隔离单模型失败并继续；`WARMUP_FAIL_FAST=true` 时首个失败即抛出。

## [0.5.46] - 2026-06-26

本版本收口 CPU-only（无 GPU / 无 CUDA）部署链路的复核问题：修复 CPU 编排被共享 `.env` 中 GPU 默认值影响的隐患，把 CPU-only 部署契约纳入部署检查与生产就绪门禁，并同步补齐部署文档、环境示例和回归测试。

### CPU-only 部署收口

- `docker-compose.cpu.yml` 固定 `FORCE_CPU: "true"`，不再通过 `${FORCE_CPU:-true}` 读取共享 `.env`，避免通用环境文件中的 `FORCE_CPU=false` 让 CPU 容器误走 GPU 探测路径。
- `docker-compose.cpu.yml` 改用独立 `CPU_TRUSTED_HOSTS` 默认值，默认包含 `127.0.0.1,localhost,cpu-worker-0,portrait-stream-worker`，避免 GPU 部署的 `TRUSTED_HOSTS=gpu-worker-0,gpu-worker-1` 覆盖 CPU 服务名。
- CPU 编排继续使用 `Dockerfile.cpu`、CPU 版 `onnxruntime==1.20.1` 和 `NVIDIA_VISIBLE_DEVICES=none`，并保持无 `gpus`/`deploy.resources.reservations.devices`/`driver: nvidia` 等 GPU reservation 配置。
- `.env.example` 新增 `CPU_TRUSTED_HOSTS` 示例，并说明 CPU compose 内部固定 `FORCE_CPU=true`，不会被共享 `.env` 的通用 `FORCE_CPU` 值覆盖。

### 部署门禁与生产就绪检查

- `tools/deploy_check.py` 将 `Dockerfile.cpu`、`docker-compose.cpu.yml`、`requirements-cpu.txt` 和 `requirements-cpu.lock` 纳入必备部署文件检查。
- 新增 CPU compose 契约检查：`cpu_compose_force_cpu_is_literal`、`cpu_compose_trusted_hosts_isolated`、`cpu_compose_has_no_gpu_reservation`、`cpu_compose_uses_cpu_dockerfile`，防止后续改动重新引入 GPU 依赖或 Host 白名单串味。
- `tools/deploy_check.py` 增加 `onnxruntime_version()` 精确包名解析和 `check_dependency_lock()`，校验 CPU/GPU 两套依赖文件中的 ONNX Runtime 版本对齐，并确保 CPU 文件不混入 `onnxruntime-gpu`、GPU 文件不误匹配 CPU 包。
- `tools/portrait_production_readiness.py --scope platform --strict` 新增 CPU-only 部署契约校验，平台级严格检查现在能覆盖 CPU compose、CPU Dockerfile 和 `.env.example` 的关键约束。

### 文档

- `README.md` 新增 CPU-only 部署说明，明确使用 `docker compose -f docker-compose.cpu.yml up -d --build`、CPU 版依赖和独立 Host 白名单。

### GPU 队列、并发与性能治理

- 模型配置可通过 `max_concurrency` 或 `runtime.max_concurrency` 覆盖单模型并发。
- `MODEL_QUEUE_TIMEOUT_SECONDS` 控制模型队列和 GPU 队列等待超时。
- 模型配置可通过 `queue_timeout_seconds` 或 `runtime.queue_timeout_seconds` 覆盖队列超时。
- 队列等待超过配置值会返回 503，避免请求无限堆积。
- 支持单请求内 batch。
- 如果模型不支持动态 batch，检测/ReID 等路径可自动降级为逐帧推理。

- 支持 `X-API-Key: <token>`。
- 模型路径使用 `Path.resolve()` 限制在模型根目录内，防止路径穿越。
- 上传图片、视频、tensor 均有数量或大小限制。
- 视频流拉取默认关闭，降低 SSRF 风险。

### 可观测性

- 所有 HTTP 请求返回 `X-Request-ID`。
- 调用方可传入 `X-Request-ID`。
- 日志使用 JSON 字符串记录关键事件。
- 响应和日志记录 request_id、model、alias、task、image_count、result_count、cold_loaded、queue_seconds、load_seconds、inference_seconds、preprocess_seconds、postprocess_seconds、total_seconds 等字段。
- `/metrics` 暴露 Prometheus 文本指标。
- 指标覆盖 HTTP 请求、predict 请求、模型加载、缓存命中、模型卸载、已加载模型数、耗时累计、业务请求、业务错误、检测人数、处理帧数、图像解码、预处理和后处理。
- 模型维度指标包括：
  - `gpu_worker_model_config_info`
  - `gpu_worker_model_loaded_info`
  - `gpu_worker_model_file_bytes`
  - `gpu_worker_model_load_count_total`
  - `gpu_worker_model_inference_count_total`
  - `gpu_worker_model_last_used_at_seconds`
- 模型指标标签包含 `model`、`task`、`version`、`status`。

### 调试能力

- 提供 `POST /debug/model-output`。
- 调试接口可上传样例图片，返回输入 shape、输出 shape、dtype、min/max 和少量 sample。
- 提供 `/ready/deep` 检查配置模型、可选加载模型、可选 dummy inference。
- 提供 `/model-info` 查看模型 session 输入输出和 provider。
- 提供 `/model-package` 查看模型包、模型卡、labels、sha256 匹配状态。

### 部署

- 使用 Docker Compose 启动两个 GPU worker。
- 默认 worker 为 `gpu-worker-0` 和 `gpu-worker-1`。
- 默认宿主机端口为 `127.0.0.1:9001` 和 `127.0.0.1:9002`。
- 默认容器服务端口为 `8000`。
- 每个 worker 可绑定不同 GPU：
  - `GPU_WORKER_0_DEVICE`
  - `GPU_WORKER_1_DEVICE`
- 容器内模型目录为 `/models`。
- 共享模型宿主机目录由 `MODELS_HOST_DIR` 配置。
- Dockerfile 基于 CUDA 11.8 runtime。
- 固定 `onnxruntime-gpu==1.18.0`。
- 设置 `TZ=Asia/Shanghai`。
- 提供 Ubuntu 部署文档 `DEPLOY_UBUNTU.md`。

### 当前边界

- 本服务不包含训练、标注、模型研发平台工程落地。
- 当前 `/infer/person-tracks` 返回检测结果和 ReID embedding，不生成最终跨帧 `track_id`。
- TensorRT/GPU 实际推理需要在具备真实模型文件、CUDA、ONNX Runtime GPU provider 和 TensorRT provider 的服务器上最终验证。
- 跨请求动态 batch/批处理窗口未默认启用，需结合真实 QPS、延迟目标和模型能力单独压测。
- 视频流拉取默认关闭，生产环境建议仅在可信内网启用。
